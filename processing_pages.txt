PART 1 — DATA MODELS
────────────────────────────────────────────────────────────────────────────────

All types must be defined in a shared types file before tests or implementation
are written.

1. BookRecord
   Fields:
   - id: string
   - name: string
   - url: string
   - category: string
   - writer: string | null
   - translator: string | null
   - composer: string | null
   - createdAt: string       // ISO timestamp
   - updatedAt: string       // ISO timestamp
   - bookCreationState: BookCreationState

   BookCreationState is one of:
   - "not_created"           // No creation request has ever been submitted
   - Any valid BookCreationRequestState (mirrors the active request's state)

2. BookCreationRequest
   Fields:
   - id: string
   - bookRecordId: string
   - state: BookCreationRequestState
   - createdAt: string
   - updatedAt: string
   - progress?: RequestProgress | null     // Only present when state = paused
   - errorMessage?: string | null          // Only present when state = failed
   - isResumed?: boolean                   // Flag set when resuming a paused request
   - isConfirmedNotDuplicate?: boolean     // Flag set when user chooses "New" on a duplicate
   - duplicateOfRequestId?: string | null  // Set when duplicate is confirmed
   - duplicateOfRecordId?: string | null   // Set when duplicate is confirmed

   BookCreationRequestState (string union):
   "initial" | "queued" | "processing" | "created" |
   "paused"  | "failed" | "duplicate"  | "deleted"

   Terminal states:   created, failed, duplicate, deleted
   Holding state:     paused  (can be resumed)
   Active states:     initial, queued, processing

   RequestProgress:
   - savedAt: string         // ISO timestamp of when progress was saved
   - checkpoint: string      // Identifier of the last successfully processed step
   - savedData: unknown      // Partial book data collected before the pause

PART 2 — GLOBAL UI RULES
────────────────────────────────────────────────────────────────────────────────

These rules apply to every card on every page. Write a shared test helper or
fixture that verifies these behaviors and reuse it across test files.

SELECTION
  - Each card has a leftmost checkbox column for selecting single or multiple
    rows.
  - When an action is in progress within a card:
    * All checkboxes and action buttons inside THAT card are disabled.
    * All other cards on the same page remain fully interactive — their
      checkboxes and buttons must NOT be disabled.

LOADERS
  - A loading indicator appears inside the card where an action is processing.
  - The loader disappears when the action completes or fails.
  - A loader in one card must not affect the display or interactivity of any
    other card.

SEARCH (per card)
  - Searchable fields: Name, Writer, Translator, Publisher.
  - Search is case-insensitive and filters results in real time.

FILTERS (per card)
  - Filterable by: Book Category, Book Creation Request Status.
  - Active filters are clearly indicated.

RECORD COUNT
  - Each card displays the count of currently visible (filtered) records.
  - The count updates reactively as search or filter values change.

DESIGN
  - Preserve the existing minimal, clean design language.
  - Reuse existing theme tokens, component styles, and layout patterns.
  - Do not introduce new design patterns unless there is no existing equivalent.

Book Creation Pipeline (automated, no user intervention required):

PART 4 — CROSS-CUTTING TEST SCENARIOS
────────────────────────────────────────────────────────────────────────────────

Write dedicated test files for these scenarios that span multiple pages:

1. Full Happy Path
   [ ] Manual sync fetches records → user selects records → clicks Create Book
       → requests appear in Create/Requests → auto-advance through queue and
       processing → appear in Create/Created.

2. Pause and Resume
   [ ] Sync is paused mid-way → progress saved → sync resumed → reconciliation
       runs → remaining records fetched → sync completes.
   [ ] Request paused during processing → appears in On Hold/Paused → resumed
       → moves back through initial/queued/processing → reaches created.

3. Failure Recovery
   [ ] Request fails during processing → appears in On Hold/Failed with error
       message → retried → restarts pipeline → reaches created.

4. Duplicate Resolution — New
   [ ] Duplicate detected → appears in On Hold/Duplicate → user selects New
       → moves to Requests → full pipeline completes without re-flagging.

5. Duplicate Resolution — Confirmed
   [ ] Duplicate detected → user confirms Duplicate → original request proceeds
       → duplicate book record is locked in Catalog until original is terminal.

6. Delete and Recreate
   [ ] Request deleted at any stage → appears in On Hold/Deleted → Create Again
       → moves to Requests → pipeline restarts.

7. Card Isolation
   [ ] Trigger a long-running action in Card A.
   [ ] Simultaneously interact with Card B (select rows, trigger a different action).
   [ ] Verify Card B's action completes successfully and Card A's loader is
       unaffected by Card B's activity.

8. Automated Sync + Auto-Request Creation
   [ ] Automated sync runs → records with "not_created", "failed", or "deleted"
       state get new "initial" requests → pipeline auto-advances to "created"
       without any user interaction.

PART 5 — IMPLEMENTATION CONSTRAINTS
────────────────────────────────────────────────────────────────────────────────

- Write tests first. No implementation code until the tests for that feature
  are written and confirmed to fail for the right reason.
- Use data-testid attributes on all elements that tests interact with or assert
  on. Follow the naming pattern: [page]-[section/card]-[element], e.g.,
  "catalog-records-table", "create-requests-card-delete-btn".
- API calls must be mockable. Use Playwright's route interception or a dedicated
  mock layer to simulate all backend responses including errors, delays, and
  pagination signals.
- The automated pipeline (initial → queued → processing → created) must be
  implemented as a background process that does not require the user to be on
  any specific page.
- Preserve all existing composables, utilities, UI components, and design tokens.
  Do not rewrite anything that is not part of the Book Processing pages.
- State management: use the project's existing store pattern. Do not introduce
  a new state management library.
- All new code must be TypeScript-strict with no implicit `any`.
